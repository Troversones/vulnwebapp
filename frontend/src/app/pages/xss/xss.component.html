Tárolt Cross-Site Scripting (Stored XSS)

A Stored XSS (más néven Persistent XSS) akkor fordul elő, amikor a rosszindulatú script tartósan eltárolódik a cél szerveren
— például egy adatbázisban, message forum-ban, comment field-ben vagy bármilyen más adattárolóban.
Amikor más felhasználók lekérik ezt a tárolt adatot, a rosszindulatú script lefut a böngészőjükben. Ez lehetővé teszi a támadók számára, hogy érzékeny információkat lopjanak el, session cookie-kat szerezzenek meg, vagy akár teljesen átvegyék az áldozat fiókját.


Erről bővebben: Portswigger Stored XSS

Hogyan működik az XSS?

1.
Injektálás: A támadó rosszindulatú JavaScript kódot küld be egy input mezőn keresztül (például egy comment form segítségével).
2.
Tárolás: Az alkalmazás elmenti ezt a rosszindulatú kódot az adatbázisába, megfelelő input sanitization (beviteli mezők tisztítása) nélkül.
3.
Betöltés: Amikor egy másik felhasználó megtekinti az érintett oldalt, az alkalmazás lekéri a tárolt adatokat az adatbázisból, és beilleszti azokat az oldal tartalmába.
4.
Végrehajtás: Az áldozat böngészője megkapja az oldalt, a scriptet pedig a megbízható site részének tekinti, ezért lefuttatja azt — ezzel a támadónak hozzáférést biztosít a user session-jéhez.

Sérülékeny kód példa

A következő példa egy egyszerű Angular komponenst mutat be, amely sérülékeny a Stored XSS támadásokkal szemben. A `comment.content` mező nincs megfelelően tisztítva vagy kódolva, így a rosszindulatú JavaScript kód végrehajtódhat, amikor a kommenteket megjelenítjük.

Hogyan előzzük meg az XSS-t?

✓
Input Validation: Mindig ellenőrizd és tisztítsd a felhasználói bemeneteket a szerveren.
✓
Output encoding: Kódold le a kimenetet, mielőtt megjeleníted a böngészőben, különösen HTML, JavaScript és URL kontextusokban.
✓
Content Security Policy: Implementálj CSP fejléceket a böngésző számára, hogy korlátozd a végrehajtható forrásokat.
✓
Használj Security Library-ket: Használj jól karbantartott könyvtárakat és framework-öket, amelyek beépített biztonsági funkciókat kínálnak.
